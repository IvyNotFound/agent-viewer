## ADR-004 — Tailwind CSS v3 → v4 Migration

**Date:** 2026-02-25 | **Status:** superseded by ADR-011 (Vuetify 3 migration) | **Author:** arch

> **⚠️ Superseded** — The project migrated to Vuetify 3 (Material Design 3) on branch `feat/material-design` before this Tailwind v4 migration was implemented. This ADR is obsolete — Tailwind CSS has been fully replaced by Vuetify 3. See ADR-011.

### Context

Current version: `tailwindcss ^3.4.1`. Tailwind CSS v4.0 released January 2025 (stable).

**Codebase state:**
- `tailwind.config.ts`: `darkMode: 'class'`, 4 custom brand colors (violet/emerald/amber/zinc), content glob `./src/renderer/**/*.{vue,ts,html}`
- `postcss.config.js`: plugin `tailwindcss: {}` + `autoprefixer: {}`
- `main.css`: `@tailwind base/components/utilities` directives
- ~186 Tailwind classes in 21 Vue components
- `ring-*` (focus:ring-1/ring-violet-500) in 8 files — uniform pattern
- `outline-none` in 8 files
- `placeholder-zinc-*` in several components
- No usage of `bg-opacity`, `text-opacity` (classes removed in v4 — not affected)

**Breaking changes applicable to the project:**

| Change | Impact | Affected Files |
|---|---|---|
| Remove `tailwind.config.ts` → `@theme` CSS | High | `tailwind.config.ts` + `main.css` |
| `@tailwind base/components/utilities` → `@import 'tailwindcss'` | Low | `main.css` (3 lines) |
| postcss plugin: `tailwindcss` → `@tailwindcss/postcss` | Low | `postcss.config.js` (1 line) |
| Brand colors → CSS variables `--color-brand-*` in `@theme` | Medium | `main.css` |
| `ring-*`: default ring 3px → 1px (but `ring-1` unchanged) | None | usage `ring-1` already explicit |
| `darkMode: 'class'` → default in v4, no config needed | None | removed with config |
| `placeholder-*`: syntax unchanged | None | — |

### Decision

**Migrate to Tailwind CSS v4.** Justification:

1. **Project in beta (0.x)**: no backward compatibility constraints
2. **Controlled breaking changes**: limited scope (3 config files, CSS variables for 4 colors), no usage of the riskiest patterns (`bg-opacity/*`, complex arbitrary `[attr]`)
3. **Real gains**: Rust compilation (Lightning CSS) — 5-10× faster builds; native CSS variables (better dark mode interop); `@import 'tailwindcss'` simplifies main.css
4. **`ring-1` explicit throughout codebase**: the default ring change (3px→1px) does not affect us

**postcss approach (not `@tailwindcss/vite`)** chosen because electron-vite does not yet natively support the Vite Tailwind v4 plugin in the multi-process config (main/preload/renderer) without additional configuration.

### Migration Plan (execution order)

**Scope: `devops` + `dev-front-vuejs`** | **Dedicated branch: `feat/tailwind-v4`**

#### Step 1 — Update dependencies

```bash
npm uninstall tailwindcss
npm install tailwindcss@^4 @tailwindcss/postcss@^4
# autoprefixer: optional (v4 includes vendor prefixes) but kept for compatibility
```

#### Step 2 — `postcss.config.js`

```js
// Before
module.exports = { plugins: { tailwindcss: {}, autoprefixer: {} } }

// After
module.exports = { plugins: { '@tailwindcss/postcss': {}, autoprefixer: {} } }
```

#### Step 3 — `src/renderer/src/assets/main.css`

```css
/* Before */
@tailwind base;
@tailwind components;
@tailwind utilities;

/* After — @import first, then @theme for config, then custom styles */
@import 'tailwindcss';

@theme {
  --color-brand-violet: #8b5cf6;
  --color-brand-emerald: #10b981;
  --color-brand-amber: #f59e0b;
  --color-brand-zinc: #71717a;
}
```

> `brand-violet`, `brand-emerald` etc. classes remain functional because v4 automatically generates utilities (`bg-brand-violet`, `text-brand-emerald`) from `--color-brand-*` variables.

#### Step 4 — Remove `tailwind.config.ts`

```bash
rm tailwind.config.ts
```

> `content` glob: v4 auto-detects files via `@import` and the project directory — no config needed for standard `.vue/.ts/.html` patterns.

> `darkMode: 'class'`: `dark:` behavior via `.dark` class is the **default** in v4 — no config required.

#### Step 5 — Visual verification

```bash
npm run dev
# Check: dark mode, brand colors, focus rings on inputs, placeholders
```

#### Step 6 — Tests + build

```bash
npm test
npm run build
```

### Modified Files (summary)

| File | Action |
|---|---|
| `package.json` | `tailwindcss ^3.4.1` → `^4`, add `@tailwindcss/postcss ^4` |
| `postcss.config.js` | `tailwindcss` → `@tailwindcss/postcss` |
| `src/renderer/src/assets/main.css` | `@tailwind *` → `@import + @theme` |
| `tailwind.config.ts` | **Removed** |

**Vue components: no modification required** — the `brand-*` mapping is preserved via `@theme`.

### Identified Risks

| Risk | Probability | Mitigation |
|---|---|---|
| electron-vite postcss v4 incompatibility | Low | Fallback: pin `@tailwindcss/postcss@4.0.x` stable |
| Auto `content` detection misses files | Low | Add `@source './src/renderer/**/*'` in main.css if classes are purged |
| Non-conforming `ring-*` behavior | Very low | All usages are explicit `focus:ring-1` |
| Brand color visual regression | Low | CSS variables `--color-brand-*` validated in v4 |

### Rollback Plan

```bash
git checkout main -- tailwind.config.ts postcss.config.js src/renderer/src/assets/main.css
npm uninstall tailwindcss @tailwindcss/postcss
npm install tailwindcss@^3.4.1
```

### Consequences

- Implementation ticket to be created by `review` → assigned to `devops` (npm) + `dev-front-vuejs` (CSS/config)
- Dedicated branch `feat/tailwind-v4` — PR to `main` after full visual validation
- `electron.vite.config.ts`: no modification expected (postcss managed automatically by Vite)

## ADR-005 — Maintaining sql.js + ReadFile for Concurrent project.db Access

**Date:** 2026-02-25 | **Status:** superseded | **Author:** arch

> **⚠️ Superseded by T1157** — The SQLite MCP server was reconfigured with WAL mode (`journal_mode=WAL`), which was the exact condition identified in "Conditions That Would Invalidate This Decision". better-sqlite3 was successfully adopted as the canonical DB access layer. See commit `4409094` and migration in `src/main/migration-runner.ts`.

### Context

The user requested evaluation of replacing `sql.js` with `better-sqlite3` for `project.db` access in `src/main/ipc.ts` and `src/main/migration.ts`.

**Documented constraint (MEMORY.md):** The SQLite MCP server keeps `project.db` permanently locked. `node-sqlite3-wasm` had already been tested and fails with this scenario.

**Lock mechanism analysis:**

`project.db` uses `journal_mode=DELETE` and `locking_mode=NORMAL` (confirmed via `PRAGMA`).

On Windows, SQLite implements its locks via `LockFileEx` on specific byte ranges:

| SQLite Lock | Scope | Compatible with readers? |
|---|---|---|
| SHARED | Byte range 0x40000000+ | Yes (multi-reader OK) |
| RESERVED | 1 specific byte | Yes but signals imminent write |
| PENDING | 1 specific byte | Blocks new SHARED |
| EXCLUSIVE | All bytes | Blocks everything |

The SQLite MCP server keeps a permanent open read/write connection. During a write (agents, sessions, tasks), it escalates to EXCLUSIVE — which blocks any concurrent `sqlite3_open_v2()` connection.

**Why `ReadFile()` bypasses this mechanism:**

`fs.readFile()` on Windows calls Win32 `ReadFile()`. `LockFileEx` implements "advisory" locks at the Windows level — they block other `LockFileEx` calls, but **not** raw `ReadFile()` accesses. sql.js loads bytes into WASM memory then processes them independently, without ever calling `LockFileEx`.

**Why `better-sqlite3` cannot bypass this mechanism:**

`better-sqlite3` is a native C++ binding that calls `sqlite3_open_v2()`. This function calls the SQLite Windows VFS, which itself calls `LockFileEx` to coordinate concurrent access. Even with `{ readonly: true }`, better-sqlite3 must acquire a SHARED lock via `LockFileEx` — which is blocked if the MCP holds a PENDING or EXCLUSIVE lock at the time of the attempt.

This behavior is **identical** to `node-sqlite3-wasm`, which also uses `sqlite3_open_v2()` (via WASM, but with the same SQLite VFS). The empirical proof of `node-sqlite3-wasm` failure validates that `better-sqlite3` would fail equivalently.

### Decision

**REJECT the migration to better-sqlite3** for `project.db` access.

**MAINTAIN** the `sql.js` + `fs.readFile()` approach as the canonical solution for concurrent access to a SQLite DB locked by another process on Windows.

### Conditions That Would Invalidate This Decision

If the SQLite MCP server is reconfigured with:
- `PRAGMA journal_mode = WAL`: WAL allows concurrent readers even during a write. In this case, `better-sqlite3 { readonly: true }` would work without conflict.

This would be a two-step migration: (1) force WAL on `project.db`, (2) replace sql.js with better-sqlite3. Out of scope as long as the MCP uses DELETE journal mode.

### Consequences

- **No code modification** — documented non-migration decision.
- `sql.js` remains the canonical dependency for DB reads in `ipc.ts` and `migration.ts`.
- **Documented residual risk**: the read-copy-update pattern (`readFile → modify → writeFile`) used for KanbAgent writes has a race condition risk if the MCP writes simultaneously. This risk is pre-existing and out of scope for this evaluation.
- Any future proposal to migrate to `better-sqlite3` or `node-sqlite3-wasm` must go through `arch` and reference this ADR.

## ADR-006 — Multi-Instance Agent Concurrency

**Date:** 2026-02-26 | **Status:** accepted | **Author:** arch

### Context

When launching N instances of the same agent (e.g. 2× `dev-front-vuejs`), the following risks exist:

1. **Double task claim** — two instances read the same `todo` ticket and both set `in_progress`
2. **File conflicts** — two instances modify the same files simultaneously
3. **Git conflicts** — modifications on the same branch/worktree → impossible merge
4. **DB write loss** — `dbw.js` does `readFile → modify → writeFile`: if two writes overlap, the last one overwrites the first (last-write-wins)

**Existing mechanisms:**
- `locks` table: `(file, agent_id, session_id)` — designed for one agent at a time, not tested in multi-instance
- `dbw.js`: non-atomic read-copy-update (ADR-005 documented residual risk)
- Session: each instance creates its own session → `session_id` naturally serves as instance identifier

### Decision

#### 1. Atomic Task Claim via Claim-and-Check

The `UPDATE ... WHERE status='todo'` pattern in `dbw.js` is pseudo-atomic (only one process writes at a time because `writeFileSync` takes a brief OS lock). But the real risk is TOCTOU: two instances **read** the ticket as `todo` via `dbq.js`, then each **writes** `in_progress`.

**Solution: claim-and-verify.**

```
# Step 1 — Claim (each instance attempts)
node scripts/dbw.js "UPDATE tasks SET status='in_progress', session_id=<MY_SESSION> WHERE id=<TASK_ID> AND status='todo'"

# Step 2 — Verify (instance checks it won)
node scripts/dbq.js "SELECT session_id FROM tasks WHERE id=<TASK_ID> AND status='in_progress'"
# If session_id ≠ MY_SESSION → task was taken by another instance → move to next ticket
```

This pattern works because:
- `writeFileSync` serializes writes at OS level (brief file lock)
- Post-write verification detects race loss: if the other instance wrote after, its `session_id` will be present
- No modification to `dbw.js` required

**Updated workflow rule:** replace `UPDATE tasks SET status='in_progress'` with the claim-and-verify pattern above in WORKFLOW.md "Step 3".

#### 2. Reinforced File Locks — Mandatory Check Before Modification

The `locks` table is already structurally correct. The `session_id` distinguishes instances.

**Reinforced protocol:**

```sql
-- Before modifying a file, check that no active lock exists for another session_id
SELECT l.file, a.name, l.session_id
FROM locks l JOIN agents a ON a.id = l.agent_id
WHERE l.file = '<file>' AND l.released_at IS NULL AND l.session_id != <MY_SESSION>;

-- If non-empty result → DO NOT modify, move to another file or wait
-- If empty → place the lock
INSERT OR REPLACE INTO locks (file, agent_id, session_id) VALUES ('<file>', :agent_id, :session_id);
```

**Known limitation:** this check-then-lock also has a minimal TOCTOU (two instances check, then lock). Acceptable because:
- The race window is a few ms (query + insert)
- In practice, dev agents work on different files (distinct scopes)
- Git isolation (point 3) is the final safety net

#### 3. Git Isolation — Worktree per Instance

Each instance works in a **dedicated git worktree**, created at session startup.

**Naming convention:**
```
Branch: agent/<agent-name>/s<session-id>
Worktree: .claude/worktrees/s<session-id>
```

**Lifecycle:**
```bash
# Creation (session start)
git worktree add .claude/worktrees/s<session_id> -b agent/<agent-name>/s<session_id>

# Work — agent operates in the worktree
cd .claude/worktrees/s<session_id>

# Cleanup (session end)
git worktree remove .claude/worktrees/s<session_id>
# Branch remains for merge/review
```

**Impact on terminal.ts:** the `cwd` passed to `terminal:create` must point to the worktree if multi-instance is enabled. Add an optional `workDir` parameter to the `terminal:create` IPC.

#### 4. UI Visibility — Instance Distinction

Sessions already have a unique `id`. To distinguish instances in the UI:

- **Sidebar / AgentLogsView**: display `agent-name #S<session_id>` when multiple sessions of the same agent are simultaneously `in_progress`
- **BoardView / TaskCard**: the task displays `session_id` in addition to `agent_name` when assigned

**No new DB column** — the existing `session_id` suffices as instance identifier.

#### 5. Merge Workflow — Review as Integrator

```
1. Instance completes its work → pushes its branch agent/<name>/s<id>
2. Instance marks its task `done`
3. review-master (or review):
   a. Reviews code on the branch
   b. Merges to main (fast-forward or merge commit depending on conflicts)
   c. If conflict → creates a resolution task assigned to the relevant agent
   d. Archives the task
4. Cleanup: review deletes the branch after merge
```

**Merge order:** by task priority (`critical > high > normal > low`), then by `completed_at` (FIFO).

### Required Modifications

| Component | Modification | Effort |
|---|---|---|
| `WORKFLOW.md` | Step 3: claim-and-verify + reinforced locks | Low |
| `CLAUDE.md` | Agent protocol section: mention multi-instance worktree | Low |
| `terminal.ts` | Optional `workDir` parameter in `terminal:create` | Low |
| `LaunchSessionModal.vue` | "multi-instance" option that creates worktree at launch | Medium |
| Sidebar/BoardView | Conditional `#S<id>` display | Low |
| No DB change | existing `session_id` suffices | None |

### Risks

| Risk | Probability | Mitigation |
|---|---|---|
| TOCTOU on claim-and-verify | Low | Race window < 100ms + post-write verify |
| Merge conflicts between worktrees | Medium | Review as integrator + resolution tasks |
| Orphaned worktrees (agent crash) | Medium | Cleanup at startup: `git worktree prune` |
| `project.db` corruption (concurrent writes) | Low | Serialized by `writeFileSync` OS-level lock |

### Out of Scope

- Migration of `dbw.js` to a transactional mechanism (WAL, better-sqlite3) — see ADR-005
- Centralized orchestrator (task queue, broker) — over-engineering for current volume (< 10 simultaneous instances)
- Auto-scaling of instance count — manual operator decision

### Consequences

- No DB migration needed — existing structures (`sessions`, `locks`, `tasks.session_id`) suffice
- The claim-and-verify pattern is backward-compatible: a solo agent uses it without side effects
- Worktree isolation is opt-in: a solo agent continues to work on `main` directly
- Implementation tickets to be created by `review`: one for WORKFLOW.md, one for `terminal.ts` workDir, one for multi-instance UI
